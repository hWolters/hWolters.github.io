<div class="legacy-content"><p>So many people have recommended Git as a version control system to me. I had a look at it, but I was pretty overwhelmed. Since I did not have a technical background, everything seemed so complex! Many tutorials let me copy paste code without giving you a deeper understanding of what and why I am actually doing this. This copy pasting feels like success at first, but when I tried working with it, I could not.</p>&#10;<p>The goal of this post is to teach you everything you need to know about Git on how use it as your version control system. We’re not going to talk about collaboration with git so far. First you need to understand some basic things.</p>&#10;<p>Git works with <strong>repositories</strong>. A repository is a folder, that contains a <code>.git</code> file.</p>&#10;<p>In git there are several directories where you save your code:</p>&#10;<ul>&#10;<li><strong>Working directory</strong>: This is where you save your files on your local file system</li>&#10;<li><strong>Stating Area (a.k.a. Index)</strong>: This is basically temporary memory. You can use this to save a file, that is not done, yet, but you reached a point where you feel the need to save something.</li>&#10;<li><strong>Local repo</strong>: This is a local copy of the repo. Everything you save there, will show up in your history.</li>&#10;<li><strong>Upstream repo</strong>: This is a remote copy of a repo. So this is not on your computer, but saved for example on Github or Gitlab.</li>&#10;</ul>&#10;<p>Every file has to pass through the different stages. By using <code>git add</code> you put a local file into the staging area. Use <code>git commit</code> to save a file to your local repo.</p>&#10;<table style="width:93%;">&#10;<colgroup>&#10;<col width="25%">&#10;<col width="68%">&#10;</colgroup>&#10;<thead>&#10;<tr class="header">&#10;<th>location</th>&#10;<th>what should I put there?</th>&#10;</tr>&#10;</thead>&#10;<tbody>&#10;<tr class="odd">&#10;<td>working directory</td>&#10;<td>anything! This is your local file system</td>&#10;</tr>&#10;<tr class="even">&#10;<td>staging area</td>&#10;<td>everything, that reached a certain milestone, but is not completely done</td>&#10;</tr>&#10;<tr class="odd">&#10;<td>local repo</td>&#10;<td>everything you want to see in your file history</td>&#10;</tr>&#10;<tr class="even">&#10;<td>upstream repo</td>&#10;<td>everything, that is part of a finished unit</td>&#10;</tr>&#10;</tbody>&#10;</table>&#10;<div class="figure">&#10;<img loading="lazy" decoding="async" src="/post/2018-12-17-everything-you-need-to-know-to-use-git-for-version-control_files/git_basics.png" alt="Illustration for Everything You Need to Know to Use Git for Version Control">&#10;&#10;</div>&#10;<p>So you have learned, that there are different stages in which you can save your files. Let’s say you want to go back to an older version of a file. You can use <code>git reset</code>. Depending on the next parameter, you will either reset the file in your working directory (<code>git reset --hard</code>), in your staging area (<code>git reset --mixed</code>) or in your local repo (<code>git reset --soft</code>) <img loading="lazy" decoding="async" src="/post/2018-12-17-everything-you-need-to-know-to-use-git-for-version-control_files/gitreset.png" alt="Illustration for Everything You Need to Know to Use Git for Version Control"></p></div>
