<div class="legacy-content"><p>In this blogpost you will get a basic understanding about message brokers. We will look at two very popular message brokers, Kafka and RabbitMQ, and learn, how they handle messages.</p>&#10;<p>Let’s start with a short intro about what message brokers are and how they are different from an API: Message brokers allow communication between applications, systems and services. On a high level, can understand message brokers as translators: They receive messages, which they “translate” and pass on to other systems.</p>&#10;<p>While for example REST APIs are designed to expect an immediate response, message brokers can communicate asynchronously. That means, a sending service does not have to wait for the reply of the receiving service.</p>&#10;<p>Let’s go more into detail. There are four components in message brokers:</p>&#10;<ul>&#10;<li>&#10;<p><strong>Producers</strong>: The senders of a message. For example, if a user creates an account on your website, a producer, let’s call hom “New User Service” creates a message, that an account has been created.</p>&#10;</li>&#10;<li>&#10;<p><strong>Consumers</strong>: The receiver of a message. Consumers looks at incoming messages and may execute some action, e.g. sending an Email to this new customer to verify his account.</p>&#10;</li>&#10;<li>&#10;<p><strong>Queues</strong>: Storage of the messages until they are consumed.</p>&#10;</li>&#10;<li>&#10;<p><strong>Exchange</strong>: A “config” for routing messages. The producer transmits a message to the <strong>exchange</strong> using a specified protocol (e.g. AMQP). The exchange makes sure to route the message to the correct queue.</p>&#10;</li>&#10;</ul>&#10;<p>The producers can issue messages without knowing where the consumers are, whether or not they are active, or how many of them exist.</p>&#10;<p>Let’s look at two different architectural approaches: RabbitMQ or Kafka Architecture;</p>&#10;<h2 id="basic-architecture-rabbitmq-amqp">Basic Architecture RabbitMQ (AMQP)</h2>&#10;<p>RabbitMQ consists of exchanges and queues. Exchanges are responsible for routing incoming messages. The queues store messages.</p>&#10;<p>A producer generates messages and pushes them into an exchange. The exchange receives the messages and routes them into a queue. The <strong>routing key</strong> is like the address of the message, so the exchange knows to which queue the message should be routed. The queues store the messages until a consumer takes a message off the queue. RabbitMQ is a general purpose message broker that supports different protocols including, MQTT, AMQP, and STOMP.</p>&#10;<p><img src="https://miro.medium.com/max/1400/1*_ucX-gnGYuKl71LsmfOmSQ.png" alt="Message Exchange RabbitMQ">&#10;<a href="https://miro.medium.com/max/1400/1*_ucX-gnGYuKl71LsmfOmSQ.png">Source</a></p>&#10;<h2 id="basic-architecture-kafka">Basic Architecture Kafka</h2>&#10;<p>Instead of pushing messages to a queue, Kafka stores messages in <strong>topics</strong>. A topic is a partitioned log of messages, where each partition is an <strong>ordered</strong>, immutable sequence of records. Kafka appends messages to the topics as they arrive. The messages stay for a specified number of time. Kafka does not care if messages are read or not. During the timeframe, consumers can <strong>persist</strong> and re-process the data as they need (Dumb broker / smart consumer approach). Because partitions are ordered, messages can be processed in batches.</p>&#10;<p><img src="https://miro.medium.com/max/1400/1*Ex6oX0h2k39fK_KPq5gVQQ.png" alt="Kafka producers">&#10;<img src="https://miro.medium.com/max/1400/1*fB3VV7HLNI9Vg1VjjLCPkQ.png" alt="Kafka consumers"></p>&#10;<p><a href="https://miro.medium.com/max/1400/1*_ucX-gnGYuKl71LsmfOmSQ.png">Source</a></p>&#10;<h2 id="summing-up">Summing up</h2>&#10;<p>RabbitMQ and Kafka use different approaches for processing messages:</p>&#10;<p>RabbitMQ uses a <strong>push model</strong>. It distributes messages individually through configurable routing and sends them to the consumers. Once consumed, the messages are gone.</p>&#10;<p>Kafka uses a <strong>pull model</strong>. Consumers have to request messages from a specific offset. Messages are retained in a partition and can be reread.</p></div>
