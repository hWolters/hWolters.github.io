<div class="legacy-content"><p>This post will teach you the inution of REST APIs and how you can use them to get interesting datasets for your data projects. First, we will look at the four components of a request. In the second part of this blogpost, we will go through one example and access the coingecko API via <code>curl</code>.</p>&#10;<p>So let’s start from 0: What is an API? API stands for <strong>a</strong>pplication <strong>p</strong>rogramming <strong>i</strong>nterface. APIs contain a set of methods that allow the communication between your computer and a server. &nbsp;In other words, they send back and forth data using HTTP requests. You can retrieve data (<code>GET</code>), send data (<code>POST</code>,<code>PUT</code>) or delete data (<code>DELETE</code>). You have probably seen the “Like on Facebook” or “Share on Twitter” buttons on various websites. When you click one of these buttons, the site you’re visiting communicates with your Facebook or Twitter account, and alters its data by adding new likes or creating a tweet. For you as a data scientist/analyst APIs provide a great opportunity to retrieve interesting datasets. For example the Kayak API let’s you access a lot of travel data while the imdb API contains movie data. Also some models like <a href="https://pypi.org/project/bert-api/">BERT</a> are accessible through an API.&nbsp;</p>&#10;<p>There are different ways and rules to follow when creating APIs. REST stands for “Representational State Transfer” and is just one option.&nbsp;A REST request can have four components:</p>&#10;<br>&#10;<ol>&#10;<li>method</li>&#10;<li>endpoint&nbsp;</li>&#10;<li>header</li>&#10;<li>body</li>&#10;</ol>&#10;<p>We’ll look into each component and then puzzle them together. After this chapter, you will understand the structure of a REST call and how to use it to get data.</p>&#10;<h3 id="1-method">1. Method&nbsp;</h3>&#10;<p>Request methods characterize what action we are going to take by referring to the API. In total, there are four main types of actions, but only the <code>GET</code> method is important when downloading data. <code>POST</code>, <code>PUT</code> and <code>DELETE</code> modify data. The following table gives an overview.</p>&#10;<br>&#10;<table>&#10;<thead>&#10;<tr>&#10;<th style="text-align:left">method</th>&#10;<th style="text-align:left">what it does</th>&#10;<th></th>&#10;</tr>&#10;</thead>&#10;<tbody>&#10;<tr>&#10;<td style="text-align:left"><code>GET</code></td>&#10;<td style="text-align:left">requests data from a server</td>&#10;<td></td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left"><code>POST</code></td>&#10;<td style="text-align:left">adds new data to the server</td>&#10;<td style="text-align:left">&nbsp;</td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left"><code>PUT</code></td>&#10;<td style="text-align:left">change existing data</td>&#10;<td style="text-align:left">&nbsp;</td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left"><code>DELETE</code></td>&#10;<td style="text-align:left">delete existing data</td>&#10;<td></td>&#10;</tr>&#10;</tbody>&#10;</table>&#10;<h3 id="2-endpoint">2. Endpoint</h3>&#10;<p>An endpoint is the Unique Resource Location (URL) where a service is accessed. So it is like a navigation that routes you to your destination. Let’s briefly talk about the different components of a URL by looking at an endpoint. One example from the twitter API for getting tweets from a specified location:</p>&#10;<br>&#10;`https://api.twitter.com/2/tweets/:ID?expansions=geo.place_id`&#10;<br>&#10;<ul>&#10;<li><strong>Scheme</strong>: identifies the protocol of the browser, e.g. <code>http</code> or <code>https</code>. The <code>:</code> is the scheme separator. The <code>//</code> establishes the start of a domain.</li>&#10;<li><strong>Domain Name, a.k.a. host</strong>: name of intended host or web server that is being requested (here: <code>api.twitter.com</code>)</li>&#10;<li><strong>Path</strong>: route to the resource you want to access. Paths are just like paths on a website. Any colons <code>(:)</code> on a path denote a variable, which you need to replace with an actual value. E.g. you replace <code>:ID</code> with your user id. This is one option to pass parameters to a rest request.</li>&#10;<li><strong>Querystring</strong>: query parameters give you the option to modify a request. A querystring begins with <code>?</code> followed by parameters and their values. Multiple parameters are separated with a <code>&amp;</code> (e.g. <code>?param1=1&amp;param2=2</code>). You find information about parameters in the API documentation.&#10;So you you know where to go, the next step is what to do.</li>&#10;</ul>&#10;<h3 id="3-header">3. Header</h3>&#10;<p>Headers contain meta information e.g. for authentication or for passing information about the body content ([see] (<a href="https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers))">https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers))</a>. In a REST request, you can add a header by typing <code>--header</code> (or short <code>-H</code>) followed by a string containing the header fields as colon-separated key-value pairs. For example if we wanted to authenticate accessing the twitter API:</p>&#10;<pre><code>curl 'https://api.twitter.com/2/tweets/:ID?expansions=geo.place_id' --header 'Authorization: Bearer $BEARER_TOKEN'&#10;</code></pre><p><code>--header</code> denotes that the header arguments start. <code>Authorization</code> is the key and <code>Bearer $BEARER_TOKEN</code> is the value (in this case the <a href="https://en.wikipedia.org/wiki/JSON_Web_Token">token</a>) we pass.</p>&#10;<h3 id="4-body">4. Body</h3>&#10;<p>The request body is used to send and receive data. So instead of passing data/parameters in the url (path parameters) or as a querystring, you can also send information in the body. The body can be in HTML, XML, JSON or other formats. You can add a body with <code>--data</code> (or short <code>-d</code>) as in this example from the twitter API.</p>&#10;<pre><code>curl --request POST &#10;  --url https://data-api.twitter.com/insights/engagement/totals &#10;  --header 'accept-encoding: gzip' &#10;  --header 'authorization: OAuth oauth_consumer_key="consumer-key-for-app",oauth_nonce="generated-nonce",oauth_signature="generated-signature",oauth_signature_method="HMAC-SHA1", oauth_timestamp="generated-timestamp",oauth_token="access-token-for-authed-user", oauth_version="1.0"' &#10;  --header 'content-type: application/json' &#10;  --data '{&#10;                "tweet_ids": [&#10;                    "1060976163948904448","1045709644067471360"&#10;                ],&#10;                "engagement_types": [&#10;                    "favorites","replies","retweets","video_views","impressions","engagements"&#10;                ],&#10;                "groupings": {&#10;                    "perTweetMetricsOwned": {&#10;                        "group_by": [&#10;                            "tweet.id",&#10;                            "engagement.type"&#10;                        ]&#10;                    }&#10;                }&#10;            }' &#10;</code></pre><h2 id="example-sending-requests-via-curl">Example: Sending requests via curl</h2>&#10;<p>There are different ways to access APIs. One quick appoach is through using the <code>curl</code> command from your terminal. You just need the keyword <code>curl</code> (which stands for <strong>c</strong>lient <strong>url</strong>) and our four components of the REST call.</p>&#10;<h3 id="anatomy-of-a-rest-call-via-curl">Anatomy of a rest call via curl</h3>&#10;<p>Let’s recapitulate the different components of a rest call. We need a method, a URL and optionally a header and a body. So you can write a request with the following options:</p>&#10;<p><code>curl</code> <code>--request</code> + method (<code>GET</code>/<code>POST</code>/<code>DELETE</code>/<code>PUT</code>) + header (<code>--header</code>) + URL (in quotes) + body (<code>--data</code>)</p>&#10;<p>You can use a long or a short version to pass options when making the request.</p>&#10;<br>&#10;<table>&#10;<thead>&#10;<tr>&#10;<th style="text-align:left">short</th>&#10;<th style="text-align:left">long</th>&#10;<th style="text-align:left">what it does</th>&#10;</tr>&#10;</thead>&#10;<tbody>&#10;<tr>&#10;<td style="text-align:left"><code>-X</code></td>&#10;<td style="text-align:left"><code> --request</code></td>&#10;<td style="text-align:left">HTTP method to be used (<strong>can be omitted????</strong>)</td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left"><code>-d</code></td>&#10;<td style="text-align:left"><code> --data</code></td>&#10;<td style="text-align:left">Data/body to be sent</td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left"><code>-H</code></td>&#10;<td style="text-align:left"><code> --header</code></td>&#10;<td style="text-align:left">Header to be sent</td>&#10;</tr>&#10;</tbody>&#10;</table>&#10;<p>Before you make your first request, check if you have curl installed:</p>&#10;<pre><code>curl -version.&#10;</code></pre><p>If you don’t, you can get it <a href="https://curl.haxx.se/download.html">here</a></p>&#10;<p>Amazing, you can start making requests to (?????) an API. Oftentimes you will need to request developer access. Don’t let that scare you off. Usually, you will be allowed to use the API a couple of days later. For this blogpost example, let’s work with the <a href="https://www.coingecko.com/api/documentations/v3">coingecko API</a>, because you can directly start using it. Coingecko is a website that contains information and metrics of the crypto currencies.</p>&#10;<p>We will start with a test request to make sure, the API server is available</p>&#10;<pre><code>curl -X GET "https://api.coingecko.com/api/v3/ping" -H "accept: application/json"&#10;</code></pre><p>returns</p>&#10;<pre><code>{&#10;	"gecko_says": "(V3) To the Moon!"&#10;}&#10;</code></pre><p>Good, looks like it is running.</p>&#10;<p>Now you can study the documentation for the data you’re interested in. Let’s say you want to get the exchange rates of your crypto currencies in the current moment.</p>&#10;<ul>&#10;<li>base url: <code>https://api.coingecko.com/api/v3/</code></li>&#10;<li>path: <code>/exchange_rates</code></li>&#10;<li>Head: <code>accept: application/json</code> (this API returns a response in json format per default)</li>&#10;</ul>&#10;<p>Wrapping this into a <code>curl</code> request:</p>&#10;<pre><code>curl -X GET "https://api.coingecko.com/api/v3/exchange_rates" -H "accept: application/json"&#10;</code></pre><p>You will get the exchange rates in json format:</p>&#10;<pre><code>{&#10;  "rates": {&#10;    "btc": {&#10;      "name": "Bitcoin",&#10;      "unit": "BTC",&#10;      "value": 1,&#10;      "type": "crypto"&#10;    },&#10;    "eth": {&#10;      "name": "Ether",&#10;      "unit": "ETH",&#10;      "value": 38.915,&#10;      "type": "crypto"&#10;    },&#10;    "ltc": {&#10;      "name": "Litecoin",&#10;      "unit": "LTC",&#10;      "value": 202.191,&#10;      "type": "crypto"&#10;    },&#10;    &#10;    ...&#10;    &#10;</code></pre><p>As a Data Scientist/Analyst you may be more interested in the historical development of a specific currency. Checking <a href="https://www.coingecko.com/api/documentations/v3#/coins/get_coins__id__history">the API documentation</a> you will find out, that you will need to pass the coin id through a path parameter and a date as a query parameter and an optional query parameter for localization.</p>&#10;<ul>&#10;<li>base url is the same as before <code>https://api.coingecko.com/api/v3/</code></li>&#10;<li>path with a placeholder for the coin id <code>/https://api.coingecko.com/api/v3/coins/{id}/history</code>. Let’s go with <code>bitcoin</code> here.</li>&#10;<li>query parameter for date as string in <code>dd-mm-yyyy</code> format</li>&#10;</ul>&#10;<p>Leads to the following request:</p>&#10;<p><code>curl -X GET "https://api.coingecko.com/api/v3/coins/bitcoin/history?date=01-01-2020" -H "accept: application/json"</code></p>&#10;<p>Which returns a lot of data about bitcoin (price, market cap, facebook likes etc.) on that date.</p>&#10;<pre><code>{&#10;  "id": "bitcoin",&#10;  "symbol": "btc",&#10;  "name": "Bitcoin",&#10;  "image": {&#10;    "thumb": "https://assets.coingecko.com/coins/images/1/thumb/bitcoin.png?1547033579",&#10;    "small": "https://assets.coingecko.com/coins/images/1/small/bitcoin.png?1547033579"&#10;  },&#10;  "market_data": {&#10;    "current_price": {&#10;      "aed": 26429.239288693556,&#10;      "ars": 430563.0457102599,&#10;      "aud": 10256.814195551682,&#10;      "bch": 35.13945542496315,&#10;      "bdt": 610699.0910671571,&#10;      "bhd": 2712.9039956563156,&#10;      &#10;      ....&#10;      &#10;</code></pre><h4 id="request-messages">Request messages</h4>&#10;<p>Nice, if everything worked well, you should have received 200 messages. This means your request has succeeded. Let’s briefly look into the different messages that you might see. Messages with a number &gt;= 400 indicate an error.</p>&#10;<br>&#10;<table>&#10;<thead>&#10;<tr>&#10;<th style="text-align:left">number</th>&#10;<th style="text-align:left">what does it mean?</th>&#10;<th style="text-align:left">example</th>&#10;</tr>&#10;</thead>&#10;<tbody>&#10;<tr>&#10;<td style="text-align:left">200+</td>&#10;<td style="text-align:left">request has succeeded</td>&#10;<td style="text-align:left">any of the above requests</td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left">300+</td>&#10;<td style="text-align:left">request is redirected to another URL</td>&#10;<td style="text-align:left">your request is redirected to from <code>https://api.coingecko.com/api/v3/</code> to an older version <code>https://api.coingecko.com/api/v2/</code></td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left">400+</td>&#10;<td style="text-align:left">an error from the client side has occurred</td>&#10;<td style="text-align:left">e.g. you have a typo one of your parameter values, e.g. in the coin id <code>/https://api.coingecko.com/api/v3/coins/bitcion/history</code></td>&#10;</tr>&#10;<tr>&#10;<td style="text-align:left">500+</td>&#10;<td style="text-align:left">an error from the server has occurred</td>&#10;<td style="text-align:left">e.g. there is a problem with the software running on the coingecko server. You can try running the request again or you will have to wait untill someone restarted the server</td>&#10;</tr>&#10;</tbody>&#10;</table>&#10;<p>Now you might be wondering, how to deal with all this json printouts in your console. One option would be, to write them into a file</p>&#10;<pre><code>curl -X GET "https://api.coingecko.com/api/v3/coins/bitcoin/history?date=01-01-2020" -H "accept: application/json" &gt;&gt; output.json&#10;</code></pre><p>There is also the very useful <code>jq</code> tool which formats and highlights JSON. If you have <a href="https://stedolan.github.io/jq/">jq</a> installed, you can just run <code>curl ... | jq</code> to format the output:</p>&#10;<pre><code>curl -X GET "https://api.coingecko.com/api/v3/coins/bitcoin/history?date=01-01-2020" -H "accept: application/json" &gt;&gt; output.json | jq&#10;</code></pre><p>Or you make your requests directly from python and write the results into python variables. You can use the <a href="https://requests.readthedocs.io/en/master/">python requests</a> package.</p></div>
