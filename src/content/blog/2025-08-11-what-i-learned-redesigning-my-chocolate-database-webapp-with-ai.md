<div class="legacy-content"><p>In summer 2020 I had a small Python web app around one of my favorite datasets: 300 chocolates that my hunsband and me had tried over the years with reviews and ratings and a machine learning model that predicts how much we may like an unknown chocolate.</p>&#10;<p>Technically, it was a SQLite database with a search interface that was a simple keyword match.</p>&#10;<p>It worked: It was also really ugly. Because I am not a frontend developer, I had happily used Streamlit. That was a good choice for getting something working quickly, but it also meant the interface looked like what it was: a table with some hacky filters on the side.</p>&#10;<p><img src="/post/what-i-learned-redesigning-my-chocolate-database-webapp-with-ai/original-table-search.png" alt="Original table-based chocolate search interface"></p>&#10;<p>And yes, I am a little embarrassed to show this screenshot to you.</p>&#10;<p>With the help of aider around 2023 and later Codex, the app had grown the usual side-project way: adding more features like more information about chocolates and an admin interface. However, the design was still terrible: emojis and lots of boxes.</p>&#10;<p>I tried to fix it with AI design prompts. “Make this more modern.” “Improve the UX.” “Use better spacing.” I also tried design-focused coding assistants and design skills. It was still an inconsistent design with lot’s of boxes. In my chocolate search resultlist every field from the SQLite Table got it’s own box.</p>&#10;<p><img src="/post/what-i-learned-redesigning-my-chocolate-database-webapp-with-ai/summer-2025-search-results.png" alt="Summer 2025 search results"></p>&#10;<p>The useful shift came when I stopped asking AI to decorate the old interface and started using <a href="https://stitch.withgoogle.com/">Google Stitch</a> to rethink what the database was for.</p>&#10;<h2 id="the-first-mistake-asking-ai-to-redesign-the-existing-app">The First Mistake: Asking AI To Redesign The Existing App</h2>&#10;<p>My first approach was obvious:</p>&#10;<blockquote>&#10;<p>Here is my existing web app. Please redesign it.</p>&#10;</blockquote>&#10;<p>This sounds sensible. It preserves the current app. It gives the AI real context. It should reduce hallucination.</p>&#10;<p>In practice, it preserved too much.</p>&#10;<p>The AI kept inheriting the old structure: mostlhy the same page hierarchy, the same mental model, the same “database table with decoration” feeling. The design was slightly better, because there were less boxes. But it was still overloaded and buggy.</p>&#10;<p><img src="/post/what-i-learned-redesigning-my-chocolate-database-webapp-with-ai/redesign-from-previous-website.png" alt="Redesign based on the previous website"></p>&#10;<p>I iterated on this for about a week. The pages got more consistent but they did not get good.</p>&#10;<p>The important lesson was: if your current UI is the problem, giving it as the main source of truth can anchor the redesign to exactly the wrong thing.</p>&#10;<h2 id="the-better-approach-describe-the-dataset-not-the-implementation">The Better Approach: Describe The Dataset, Not The Implementation</h2>&#10;<p>The second approach was more useful:</p>&#10;<blockquote>&#10;<p>Forget the existing app. Design a small chocolate discovery website around this dataset.</p>&#10;</blockquote>&#10;<p>Instead of feeding Google Stitch my old UI, I described what the data contained and what I wanted a visitor to do:</p>&#10;<ul>&#10;<li>search chocolates</li>&#10;<li>browse a small database</li>&#10;<li>compare ratings</li>&#10;<li>inspect details, tasting notes, and metadata</li>&#10;<li>make the site feel more like discovery than administration</li>&#10;</ul>&#10;<p>You do not even have to write this feature list yourself. A surprisingly useful step is to ask an AI:</p>&#10;<blockquote>&#10;<p>Given this app and dataset, what are the core user journeys? What would make this useful and delightful?</p>&#10;</blockquote>&#10;<p>Then review the answer critically.</p>&#10;<p>This produced a much better first draft. The design finally stopped looking like a web form around a CSV file. It became closer to a small data experience: more visual hierarchy, better emphasis on discovery, clearer entry points, and fewer meaningless containers.</p>&#10;<p><img src="/post/what-i-learned-redesigning-my-chocolate-database-webapp-with-ai/redesigned-search-results-with-stitch.png" alt="Redesigned search results with Stitch"></p>&#10;<p>That first Stitch draft became the basis for the current website.</p>&#10;<h2 id="playwright-became-my-design-review-loop">Playwright Became My Design Review Loop</h2>&#10;<p>The next problem was implementation.</p>&#10;<p>AI-generated UI often looks good in the screenshot and then breaks in the browser. Text wraps badly. Cards become uneven. Mobile layouts collapse. Spacing changes when real data arrives.</p>&#10;<p>So I used Playwright as a design review loop.</p>&#10;<p>Not in a fancy way. I used it to open the page, take screenshots, compare desktop and mobile states, and catch the obvious regressions that I would otherwise miss while staring at code. This mattered because the goal was not “does the CSS compile?” The goal was “does this still look like the design I accepted?”</p>&#10;<p>This is one of my biggest learnings from the project:</p>&#10;<blockquote>&#10;<p>For AI-built frontends, screenshots are tests.</p>&#10;</blockquote>&#10;<p>Unit tests will not tell you that your beautiful card grid now has one item with a three-line title pushing everything out of alignment. A browser check will.</p>&#10;<h2 id="then-i-threw-away-the-backend-too">Then I Threw Away The Backend Too</h2>&#10;<p>After the frontend redesign worked, I became less attached to the backend.</p>&#10;<p>The old backend had accumulated along with the old UI. I decided to rebuild it from scratch as well, but with one important constraint: releases had to pass a small set of tests.</p>&#10;<p>The stack stayed boring: Python, FastAPI, SQLAlchemy, Postgres, Docker. The AI wrote a lot of code I would not have written by hand as quickly.&#10;But I understood the contract:</p>&#10;<ul>&#10;<li>the database must initialize</li>&#10;<li>the app must start</li>&#10;<li>keyword search must return expected chocolates for known queries</li>&#10;<li>important pages must render</li>&#10;<li>detail pages must keep showing the right metadata and reviews</li>&#10;<li>smoke tests must pass before release</li>&#10;</ul>&#10;<p>That distinction matters. I do not need to lovingly understand every line of framework glue. I do need to know what behavior must never break.</p>&#10;<h2 id="the-uncomfortable-part-you-can-use-a-stack-you-do-not-know">The Uncomfortable Part: You Can Use A Stack You Do Not Know</h2>&#10;<p>This project changed my opinion on using unfamiliar tech stacks.</p>&#10;<p>Previously, I would have said: do not ship code you do not understand.</p>&#10;<p>I still mostly believe that. But now I would phrase it differently:</p>&#10;<blockquote>&#10;<p>Do not ship behavior you cannot verify.</p>&#10;</blockquote>&#10;<p>There is a difference.</p>&#10;<p>If AI chooses a framework or library I would not have chosen myself, that is not automatically a problem. If I cannot test whether database access, search, error handling, and page rendering still work, that is a problem.</p>&#10;<p>AI lets you move faster into unfamiliar territory. Testing is what stops that from becoming random wandering.</p>&#10;<h2 id="the-main-lessons">The Main Lessons</h2>&#10;<h3 id="1-redesigning-from-the-old-ui-can-preserve-the-wrong-assumptions">1. Redesigning from the old UI can preserve the wrong assumptions</h3>&#10;<p>If the current app is ugly because the underlying mental model is wrong, do not start with screenshots of the current app. Start with the data and the user journeys.</p>&#10;<h3 id="2-design-tools-are-better-at-experience-shape-than-coding-agents-are">2. Design tools are better at experience shape than coding agents are</h3>&#10;<p>Coding agents can improve CSS, but dedicated design tools like Stitch gave me much better first principles for layout, hierarchy, and visual direction.</p>&#10;<h3 id="3-screenshots-are-part-of-the-test-suite-now">3. Screenshots are part of the test suite now</h3>&#10;<p>For frontend work, Playwright is not only an end-to-end testing tool. It is a way to keep the visual loop honest.</p>&#10;<h3 id="4-ai-generated-backend-code-is-acceptable-only-when-contracts-are-explicit">4. AI-generated backend code is acceptable only when contracts are explicit</h3>&#10;<p>If you let AI rebuild backend code, define the tests first. The tests are your understanding made executable.</p>&#10;<h3 id="5-the-best-prompt-is-often-not-improve-this">5. The best prompt is often not “improve this”</h3>&#10;<p>The better prompt is:</p>&#10;<blockquote>&#10;<p>Here is the goal. Here are the users. Here are the core workflows. Ignore my current implementation if it gets in the way.</p>&#10;</blockquote>&#10;<p>That shift made the biggest difference.</p>&#10;<h2 id="conclusion">Conclusion</h2>&#10;<p>The surprising thing was not that AI could make my app prettier. I expected that.</p>&#10;<p>The surprising thing was that the best result came from being willing to throw more away.</p>&#10;<p>My old chocolate database app worked, but its structure kept pulling every redesign back toward the same boring interface. Once I described the dataset and the experience I wanted instead of the app I had, the AI became much more useful.</p>&#10;<p>So my current rule is:</p>&#10;<blockquote>&#10;<p>Use AI to explore the experience from scratch. Use tests to keep the implementation grounded.</p>&#10;</blockquote>&#10;<p>That combination worked much better than asking AI to decorate old boxes.</p>&#10;<p>An in case you’re curious how this website looks like today: <a href="https://kakaokunde.vercel.app">Kakaokunde</a>. The design and website is far from perfect.</p></div>
